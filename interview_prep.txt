ISSUE 1 — FAN-OUT (★ headline)
  Naïve market+date join exploded 4,732 → 28,276 rows (6× fan-out)
  Cause: multiple events on same market+date duplicated OTB rows
  Impact if unfixed: systematic overstating of event impact
  Fix: pre-aggregate events to 841 event-day summaries (sum visitors,
       count events) BEFORE joining. Result: clean 1:1 match.

ISSUE 2 — OCCUPANCY OUTLIERS (32 rows)

  Fix: capped at 1.0 occupancy for model safety; flagged dates for ops review

  Verdict: within rounding tolerance ($1.50)

ISSUE 4 — JOIN KEY INTEGRITY (PASSED)

  No orphaned records; cancellations filtered; no duplicate event IDs

ISSUE 5 — OTB STRUCTURE CONFIRMATION

PART 9 — Q&A DEFENSE DRILLS (ANTICIPATED QUESTIONS + YOUR ANSWERS)
================================================================================

Q1 (Huan Tran — the AI audit question):
"How do you know the AI didn't write buggy logic?"

A: "I never assume it didn't. That's why I built the /reconcile-data skill
into the workflow. At OMEL AI, my entire product was about catching AI
hallucinations. I applied the same principle here. I don't read every line
of syntax — I test the OUTPUTS against strict business boundaries:
occupancy must be 0–100%, total rooms must balance, revenue must equal
rooms_sold × ADR within tolerance. If the output fails the test, the
agent rewrites it."

Q2 (Matthew Klein — scale question):
"How would you scale this across Pebblebrook's full portfolio?"

A: "The code is already modular, not a monolithic script. To scale, I'd
wrap these functions into REST endpoints using FastAPI — I have experience
with that from Softeleven. We could containerize it with Docker, run it
nightly via GitHub Actions, and feed Tableau directly. Adding a new hotel
is a config change, not a rebuild."

Q3: "Why not use a machine learning model?"

A: "With 4,732 rows across 26 snapshots and 2 hotels, we don't have the
training data to support a supervised ML model without overfitting. More
importantly, stakeholders in Finance and Operations need to be able to
ask 'why is July 11th at 100%?' and get an auditable answer. A transparent
formula beats an overfit model in this data environment — and it's far
more defensible with non-technical stakeholders."

Q4: "How confident are you in the ADR assumption?"

A: "ADR is pulled directly from current OTB. I'm not forecasting rate —
only occupancy. Rate strategy is a decision for the revenue team. The
model surfaces WHERE to act on rate; it doesn't dictate the rate itself."

Q5: "What if an event gets cancelled?"

A: "The model re-runs with updated event data. Because the pipeline is
modular, re-forecasting a single date is straightforward. This is a known
limitation I've called out explicitly — I'd rather flag it than hide it."

Q6: "How do we validate the pickup rate assumptions?"

A: "The pickup rates are derived from the OTB snapshot history in the
source data itself — not external assumptions. As more snapshots
accumulate, these rates can be recalibrated. They are empirical
measurements, not guesses."

Q7: "You used AI to write this code. Can you actually explain the logic?"

A: "The syntax was typed by an agent, but the business logic and
architecture were entirely mine. I specified: use an explainable model,
weight events by visitor density relative to room count, cap the event
lift at 15% occupancy, derive pickup rates empirically. I designed the
pipeline; the agent wrote the pandas. That distinction is the whole
point of the JD — architectural problem-solving, not syntax-level coding."
